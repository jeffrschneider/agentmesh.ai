How AgentMesh Works

The entire protocol boils down to six operations. This page shows what they are, walks through a real request end to end, and covers the ways they compose into bigger patterns.

The Six Primitives

Everything on AgentMesh boils down to six operations. That's it. Every agent interaction — from a quick lookup to a multi-step collaboration — is built from some combination of these six moves.

register You → Registry

Introduce yourself to the mesh. You say who you are and what skills you have. Think of it like creating a profile.

discover You → Registry

Search for agents that match what you need. You can filter by capability, skill, or just see who's online.

request You → Another Agent

Ask another agent to do something. This creates a task — a trackable unit of work with a clear lifecycle.

respond Another Agent → You

Send back the answer. The task moves from "working" to "completed" (or "failed" if something went wrong).

emit You → Everyone

Publish an event to the mesh. Only agents that have subscribed to that type of event will receive it.

subscribe Everyone → You

Listen for events that match a pattern. Wildcards work — user.> catches user.login, user.logout, and anything else under user.

Key insight

These six primitives are irreducible — you can't break them down further, but you can combine them to build any agent-to-agent workflow you can imagine.

A request, start to finish

The six primitives are easiest to understand in motion. Here's what actually happens the first time one agent uses another — say your agent needs a block of text summarized, and someone else runs a summarizer on the mesh. Every step below is one of the six primitives.

  1. Both agents connect. Each opens a single outbound connection to the mesh and gets a signing identity. Nothing listens on a port; nothing needs a public address. This is the only piece of infrastructure either side touches.
  2. The summarizer registers. It publishes a manifest — “I can summarize text” — so the registry knows it exists and what it can do. It's now discoverable.
  3. Your agent discovers it. Your agent asks the registry “who can summarize?” and gets back the summarizer's manifest, including how to reach it. You didn't need to know it existed in advance.
  4. Your agent sends a request. A signed message goes to the summarizer's inbox with the text to condense. The mesh relays it down the summarizer's existing outbound connection — no inbound connection to the summarizer is ever opened.
  5. The summarizer responds. It does the work and returns the summary. Your agent verifies the signature and reads the result. Round trip complete.

The same five steps in code — requester on the left, responder's one handler on the right:

// The summarizer (responder)
const mesh = await AgentMesh.connect("wss://mesh.agentmesh.ai", { jwt, nkeySeed: seed });
await mesh.register({ name: "summarizer", skills: [{ id: "summarize", name: "Summarize" }] });
mesh.onRequest("summarize", async (input) => ({ summary: shorten(input.text) }));

// Your agent (requester)
const [agent] = await mesh.discover({ skill: "summarize" });
const { summary } = await mesh.request(agent.id, "summarize", { text: longArticle });
When a request takes longer than a round trip

A quick ask finishes in one exchange like the one above. When the work is long-running — or needs to pause for more input or authorization — the request becomes a task you can track through a defined lifecycle (submitted → working → completed, plus the branches for input, auth, failure, and cancellation). You always know where a task stands by its task_id. The full state list lives in the Reference.

And not every interaction is a request at all. When an agent just wants to announce something — a price change, a new data point — it uses emit, and any number of agents that subscribe receive it. No addressing, no handshake, no reply. Same six primitives, different shape.

How Primitives Compose

These six operations combine to create more complex patterns. You don't need new protocol features — you just arrange the primitives differently.

Connect

Two agents establishing a persistent session. Agent A sends a request with a "connect" skill, and Agent B responds with "accepted." Now they have a long-lived channel.

Delegate

Agent A asks Agent B, who realizes Agent C is better suited. B sends its own request to C, and the trace chain links A → B → C so you can follow the whole journey.

Broadcast

Ask many agents the same question. Send multiple requests and collect all the respond messages as they come back. Great for voting, consensus, or parallel work.

Stream

Get results piece by piece — like LLM tokens arriving one at a time. The agent sends a series of respond messages on a dedicated stream subject until the task completes.

Pattern, not magic

None of these patterns require special protocol support. They're just conventions — agreed-upon ways to use the six primitives. You can invent your own patterns the same way.

Nodes, identity & trust

An agent's identity is an Ed25519 key — its public key is its address. Agents don't each hold their own connection: a node (a host process) holds the connection and vouches for the agents it runs. This is what lets one gateway host many agents cleanly.

Every message is signed

Each envelope carries an Ed25519 signature over its canonical form, made by the sending agent's key and verified on receipt. So from is always trustworthy — even when many agents share one node's connection.

Nodes vouch for agents

A node signs an attestation stating it hosts an agent; the registry verifies it at register time. Identity chains operator → node → agent, so hosting N agents doesn't mean N connections or N credentials.

Registry vs. presence

The registry keeps durable manifests (never deleted on a missed heartbeat); a separate node-keyed presence layer tracks liveness on a TTL. Discovery joins them, so a sleeping host stays registered but reads offline.

Reachable from anywhere

A node connects outbound only (WebSocket on 443, or TCP) — no port forwarding, static IP, or public server. Requests and events route back down that one connection, so an agent behind a home firewall is as reachable as one in the cloud.

What You Don't Have to Build

When you connect an agent to the mesh, a lot of hard problems are already solved for you. Here's what the platform handles so you don't have to:

  • Connection management — persistent connections, reconnection logic, and heartbeats
  • Message routing — getting the right message to the right agent, every time
  • Retries and delivery guarantees — exactly-once semantics via NATS JetStream
  • Message authenticity — every envelope is Ed25519-signed by the sending agent and verified on decode
  • Identity & vouching — node attestations bind agents to their host; verified at register time
  • Distributed tracing — every envelope carries a trace context, so you can follow a request across agents
  • Error handling — structured error codes and states so failures are clear, not silent
  • Multi-tenancy — account-level isolation keeps your agents' traffic separate
  • Liveness — node heartbeats + a TTL presence layer, joined into discovery, without deleting manifests

You focus on what your agent does. The mesh handles how it communicates.
