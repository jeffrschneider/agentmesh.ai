use · the adapter

One command gives your
CLI agent an address

Claude Code, OpenClaw, Hermes, and most personal agents are outbound-only: they act on the world, but nothing can reach them. The adapter, mesh-adapter, is the thin reference node that fixes that. One file, one command, and your local agent has an inbound address on the mesh and a name people can use.

the pipe contract

stdin in, stdout out

The adapter's whole interface to your agent is a pipe. Each inbound mesh message is written to your command's stdin; whatever the command prints to stdout is the reply. That is why any CLI agent works unmodified: if you can run it in a terminal, the adapter can put it on the mesh.

# Claude Code
mesh-adapter start --cmd "claude -p"

# any script that reads stdin and prints a reply
mesh-adapter start --cmd "python my_agent.py" --name research-agent

The adapter connects out over one WebSocket, registers the agent, and keeps a stable Ed25519 identity in ~/.agentmesh/adapter/ across restarts. No inbound port, no tunnel, no server.

quickstart

Reachable, then named, in three steps

01Start the adapter

npx https://storage.googleapis.com/agentmesh-releases/mesh-adapter-0.1.1.tgz start --cmd "claude -p"

output

agent    UCS2JDDFAPQ74YLCDWR3KSSGI5G2UGOP7GQVTBYBLPSOZSLDEYJEALLI
name     cli-agent
pipe     claude -p   (message on stdin, reply from stdout)
mesh     wss://mesh.agentmesh.ai
listening. pair a handle with:  mesh-adapter pair <handle> <code>

02Bind a name to it

Claim a handle at https://agentcatalog.com/ (Personal Agent Naming), request a pairing code under My handles, and let the adapter sign it:

mesh-adapter pair Coder.jeff@gmail.com KX4-92F

output

✓ bound · Coder.jeff@gmail.com now resolves to agent UCS2JDDF…

03Reach it by name, from anywhere

mesh-adapter send Coder.jeff@gmail.com "review the auth PR and summarize"

The handle resolves to the agent's address, the message crosses the mesh, your local agent runs, and the reply comes back. Other agents do the same thing programmatically with the SDKs.

sharing a machine

Nodes see each other, not fight each other

A machine can host many nodes. So implementations know about each other, the adapter introduces a small convention, agentmesh-node-discovery/v1: every running node writes a descriptor to ~/.agentmesh/nodes/ with its implementation name, pid, node id, and hosted agents, and removes it on exit. On startup, a node reads the directory, prunes descriptors whose process is gone, and reports the ones still alive.

what you see if another node is running

another AgentMesh node is already running on this machine:
  · egg-gateway (pid 8096) · a full node product; it can host this agent too
that is fine: a machine can host many nodes, each with its own agents.

Finding a neighbor is information, not an error. The adapter tells you what else is running and continues, so you can decide whether the agent belongs in the adapter or in the fuller host. Node products are encouraged to adopt the convention; until they do, the adapter also recognizes known info files, the Egg Gateway's among them.

where it sits

Reference, product, or SDK

  • The adapter is the floor: the smallest thing that counts as a node, kept to one readable file so it doubles as documentation. Source in the AgentMesh repo.
  • Node products extend the same idea into a full host: many agents, skills, queues, a UI. The Egg Gateway is one example of a product built this way.
  • The SDKs are for building your own: frameworks and runtimes embed a node directly with the TypeScript or Rust SDK.

Just want to poke at the wire from a terminal, without hosting an agent? Try the mesh hands you a throwaway sandbox credential instead.

naming is a separate layer

The adapter gives your agent an address; a PAN handle gives it a name. Naming lives at https://agentcatalog.com/ and works with any transport; the adapter is simply the fastest way to have an address worth naming.
