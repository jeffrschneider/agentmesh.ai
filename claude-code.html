Prompt 1: Build an Agent

Connect to the mesh, register skills, discover agents, send requests.

The MAPI spec

Point Claude Code at the full protocol spec so it builds correct agents.

What are agentic coding tools?

Agentic coding tools are AI-powered development environments that go beyond autocomplete. They can read your codebase, write multi-file implementations, run commands, fix errors, and iterate until things work. Think of them as a developer that lives inside your terminal or editor.

Claude Code is Anthropic's CLI-based agentic coding tool. You describe what you want in plain English, and it writes the code, runs it, and handles the back-and-forth until the task is done. It's particularly good at working with structured protocols like AgentMesh because the protocol is well-defined and the SDK provides clear TypeScript types.

The workflow

Here's what it looks like to build an AgentMesh agent with Claude Code. You don't write the networking code. You describe the agent's purpose, and Claude Code handles the rest.

Install the SDK

Tell Claude Code to set up a new project with the AgentMesh SDK. It runs npm init, installs the package, and configures TypeScript.

prompt 
Create a new TypeScript project and install the agentmesh SDK

Describe your agent

Tell Claude Code what your agent should do. Be specific about the skills it offers and how it should handle requests.

prompt 
Build an agent that connects to ws://34.58.99.231:4443,
registers with a "summarize" skill, and when it receives
a request, summarizes the input text in 2-3 sentences.

Claude Code writes the agent

It generates a complete agent: connection setup, registration, request handler, error handling, and graceful shutdown. The SDK's TypeScript types guide it to produce correct envelope structures and use the right primitives.

Run and iterate

Claude Code runs the agent, watches for errors, and fixes them. If the agent fails to connect or a type is wrong, it reads the error output and adjusts the code automatically.

Add more capabilities

Once the basic agent works, you can layer on more: "Now discover all agents with the translation capability and send them a request" or "Subscribe to document.uploaded events and summarize each one."

The MAPI protocol spec

AgentMesh publishes its complete protocol interface as a MAPI (Markdown API) file, a structured markdown document that LLMs parse natively. The file describes every primitive, every type, every error code, and every subject pattern in a format that Claude Code can read and reason about.

agentmesh.mapi.md

The complete AgentMesh protocol, all six primitives, the envelope schema, task lifecycle, 16 error codes, and every TypeScript type, in a single file that LLMs understand natively.

View the spec GitHub

When you give Claude Code the MAPI file, it doesn't need to guess how the protocol works. It reads the type definitions, understands the envelope structure, knows the valid task states, and generates code that conforms to the spec. No hallucinated field names, no wrong message types.

Try it

Paste this into Claude Code: Read docs/agentmesh.mapi.md and build me an agent that... It will use the spec to produce correct, type-safe code on the first try.

The MAPI file covers:

  • Global types: Envelope, Manifest, Skill, Task, TraceContext, and all other shared interfaces
  • All six primitives: register, discover, request, respond, emit, and inbound request handling, each with input/output types and NATS subjects
  • Task lifecycle: the full state machine with all valid transitions
  • Subscriptions: mesh events and task updates with wildcard patterns
  • Error codes: all 22 structured error codes with retry guidance
  • Subject namespace: the complete routing map

Example: An agent built by Claude Code

Here's a complete agent that Claude Code would generate when asked to build a translation agent. It registers with the mesh, handles incoming translation requests, and discovers other agents to delegate work.

typescript 
import { AgentMesh } from "agentmesh";

const mesh = await AgentMesh.connect("ws://34.58.99.231:4443");

// Register as a translation agent
await mesh.register({
  name: "Translator",
  description: "Translates text between languages",
  capabilities: ["translation"],
  skills: [{
    id: "translate",
    name: "Translate",
    description: "Translate text to a target language",
    input_modes: ["text/plain"],
    output_modes: ["text/plain"],
  }],
});

// Handle incoming translation requests
mesh.onRequest("translate", async (payload) => {
  const { text, target_lang } = payload.input as {
    text: string;
    target_lang: string;
  };

  // Your translation logic here
  const translated = await translateText(text, target_lang);

  return { text: translated, language: target_lang };
});

// Discover and use other agents
const { agents } = await mesh.discover({
  capabilities: ["spell-check"],
});

if (agents.length > 0) {
  const result = await mesh.request(agents[0].id, "check", {
    text: "Check this translated text",
  });
  console.log("Spell check:", result.payload.output);
}

// React to mesh events
mesh.subscribe("document.>", (event) => {
  console.log(`Event: ${event.event_type}`, event.data);
});

console.log("Translator agent online");

This agent uses five of the six primitives: register, discover, request, respond (via the onRequest handler), and subscribe. Claude Code generates all of this from a one-sentence prompt plus the MAPI spec.

Prompt 2: Add Presence to your app

It's an application on top of the mesh, not part of the protocol. The prompt below is historical; for protocol-level liveness see Presence & liveness.

Once your app is connected to the mesh, you can add user-to-user communication with the Presence service. This prompt gives Claude Code everything it needs: contacts, messaging, skill routing, and online/offline status.

For the full human-readable guide, see Presence.

prompt 
Add AgentMesh Presence to this app using the agentmesh SDK.
All mesh communication goes through the SDK — do NOT use raw nats.ws.

## Setup

npm install agentmesh

## Account Creation & Authentication

YOUR APP owns account creation. AgentMesh does not pre-create accounts.
When a user opens your app, you call POST /v1/connect — this creates
the account (if new), triggers email verification, and returns creds.

  // 1. Initiate connection — this creates the account if it doesn't exist
  const resp = await fetch("<AUTH_URL>/v1/connect", {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ email: user.email, appName: "YourApp" }),
  });
  const data = await resp.json();

  // 2. Handle the response — there are two possible outcomes:
  //
  // FIRST TIME (status 202): account created, email verification required
  //   { status: "verification_required", accountId: "...",
  //     message: "Check your email for the verification link." }
  //
  //   → Show the user "check your email" UI
  //   → Poll /v1/connect/status until verified:
  //
  //   const poll = await fetch("<AUTH_URL>/v1/connect/status", {
  //     method: "POST",
  //     headers: { "Content-Type": "application/json" },
  //     body: JSON.stringify({ accountId: data.accountId, appName: "YourApp" }),
  //   });
  //   // Once verified, this returns the same "connected" response below.
  //
  // RETURNING USER (status 200): already verified, creds returned immediately
  //   { status: "connected", jwt, seed, publicKey, accountId, appId, tier }

  const { jwt, seed, accountId, appId } = data; // or from poll response

  // 3. Connect to the mesh using the SDK
  import { AgentMesh } from "agentmesh";
  const mesh = await AgentMesh.connect("ws://<NATS_URL>:4443", {
    jwt, nkeySeed: seed,
  });

  // 4. Register as an agent with the skills your app handles
  await mesh.register({
    name: "YourApp",
    skills: [{ id: "your.skill", name: "Your Skill", description: "..." }],
  });

## Presence Lifecycle

All presence calls use mesh.request() to the presence service.
The service ID is "mesh.service.presence".

On startup — set presence and register as a backend:

  await mesh.request("mesh.service.presence", "mesh.presence.set", {
    accountId, email: user.email, status: "online", displayName: user.name,
  });

  await mesh.request("mesh.service.presence", "mesh.presence.backend.register", {
    accountId, appId, agentId: mesh.id,
    skills: ["your.skill.id1", "your.skill.id2"],
  });

On shutdown / logout:

  await mesh.request("mesh.service.presence", "mesh.presence.set", {
    accountId, email: user.email, status: "offline",
  });

## Contacts (Auth HTTP API — standard fetch)

Contacts are stored ON THE MESH, not locally. Do NOT create a local
database, SQLite table, or in-memory store for contacts. The mesh
is your contacts storage. Call these endpoints for every operation.

Adding a contact sends a CONNECTION REQUEST — it does NOT immediately
create a mutual contact. The other person must accept first.

  // Send a connection request
  POST   <AUTH_URL>/v1/accounts/{id}/contacts        {email}
  Response: { ok, contact, status: "pending_outgoing", invited?: true }
  // If the email is not on AgentMesh, an invitation email is sent.

  // Accept an incoming connection request (both sides become accepted)
  POST   <AUTH_URL>/v1/accounts/{id}/contacts/accept  {email}

  // Deny a connection request (silent — sender is not notified)
  POST   <AUTH_URL>/v1/accounts/{id}/contacts/deny    {email}

  // List contacts — returns live presence + connection status
  GET    <AUTH_URL>/v1/accounts/{id}/contacts
  Response: {
    contacts: [...],           // all contacts
    pending_incoming: [...],   // requests waiting for YOUR response
    pending_outgoing: [...],   // requests YOU sent, waiting for theirs
    accepted: [...]            // mutual connections (can message/route)
  }

  // Remove a contact (removed from both sides)
  DELETE <AUTH_URL>/v1/accounts/{id}/contacts/{email}

Each contact has: email, displayName, status (connection state:
pending_outgoing / pending_incoming / accepted), presenceStatus
(online/offline/away), blocked.

IMPORTANT: Messaging and routing ONLY work between accepted contacts.
Pending contacts cannot exchange messages or route skill requests.
You will get a PRESENCE_NOT_CONNECTED error if you try.

Build your UI to:
- Show incoming requests (pending_incoming) with Accept / Deny buttons
- Show outgoing requests (pending_outgoing) as "pending" state
- Only enable messaging/routing for accepted contacts
- Subscribe to contact events for real-time updates (see below)

## Contact Events (real-time notifications)

The Presence service publishes per-user contact events. Subscribe to
receive push notifications instead of polling:

  // Subscribe to your own contact events
  mesh.subscribe(`presence.contacts.${accountId}`, (payload) => {
    // payload.event_type: "contact_request" | "contact_accepted" |
    //                     "contact_denied"  | "contact_removed"
    // payload.data: { accountId, email, displayName? }
  });

Events and when they fire:
  contact_request  — someone sent you a connection request
  contact_accepted — your outgoing request was accepted
  contact_denied   — your outgoing request was denied
  contact_removed  — a contact removed you

NATS subject: mesh.event.presence.contacts.{accountId}

## Messaging (via mesh.request)

  // Send a direct message (queued if recipient is offline)
  await mesh.request("mesh.service.presence", "mesh.presence.send", {
    toEmail: "friend@example.com",
    content: { text: "Hello!" },
    messageType: "dm",  // "dm" | "voicemail" | "notification"
    fromAccountId: accountId,
    fromEmail: user.email,
  });
  // → { ok: true, messageId: "..." }

  // Get message history with a contact
  await mesh.request("mesh.service.presence", "mesh.presence.messages.list", {
    accountId, contactEmail: "friend@example.com", limit: 50,
  });
  // → { messages: [...], total: N }

  // Check unread queue depth
  await mesh.request("mesh.service.presence", "mesh.presence.messages.depth", {
    accountId,
  });
  // → { depth: 3 }

  // Drain queued messages (marks delivered, returns them)
  await mesh.request("mesh.service.presence", "mesh.presence.messages.drain", {
    accountId,
  });
  // → { messages: [...], total: 3 }

## Message Events (real-time notifications)

Subscribe to per-user message events for incoming messages and
delivery receipts:

  mesh.subscribe(`presence.messages.${accountId}`, (payload) => {
    // payload.event_type: "message_received" | "message_delivered"
    //
    // message_received:
    //   data: { messageId, fromAccountId, fromEmail, messageType, ts }
    //
    // message_delivered (delivery receipt):
    //   data: { messageId, toAccountId, toEmail, deliveredAt }
  });

NATS subject: mesh.event.presence.messages.{accountId}

Auto-drain: When a user transitions to "online", the service
automatically drains queued messages and emits delivery receipts.

## Routing Skill Requests to Other Users

  // Ask another user's app to do something
  const result = await mesh.request(
    "mesh.service.presence", "mesh.presence.route", {
      toEmail: "friend@example.com",
      skill: "browse.summarize",
      input: { url: "https://example.com/article" },
      fromAccountId: accountId,
      fromEmail: user.email,
    },
  );
  // If online: { status: "delivered", result: { summary: "..." } }
  // If offline: { status: "queued", messageId: "..." }

## Handling Incoming Routed Requests

  Incoming routed requests arrive as normal skill requests via the SDK.
  Handle them with mesh.onRequest() — no special presence code needed:

  mesh.onRequest("browse.summarize", async (input) => {
    const { url } = input;
    const summary = await summarizePage(url);
    return { summary };
  });

## Blocking

  await mesh.request("mesh.service.presence", "mesh.presence.contacts.block", {
    accountId, email: "spammer@example.com", blocked: true,
  });
  // Unblock: blocked: false

## Key Design Notes

- YOUR APP creates accounts by calling POST /v1/connect. AgentMesh does NOT pre-create them.
- First-time users get a verification email. Handle the "verification_required" flow.
- Use the agentmesh SDK for everything. Do NOT use raw nats.ws.
- Contacts are stored ON THE MESH. Do NOT create a local database for contacts.
- Contacts use a CONNECTION REQUEST flow: add → pending → accept/deny → accepted.
- Messaging and routing ONLY work between accepted contacts. Pending = no communication.
- Denied requests are silent (sender not notified) with a 1-year cooldown.
- Contacts are managed via HTTP (auth API). Everything else via mesh.request().
- Email addresses are used for addressing but are NOT discoverable.
- Messages to offline users are queued. Auto-drained when user comes online.
- Messages support types: "dm", "voicemail", "notification".
- Delivery receipts: subscribe to presence.messages.{accountId} for message_delivered events.
- Delivered messages expire after 30 days (message TTL).
- Stale users auto-set to offline after 90s (liveness sweep).
- Multiple apps can be backends for the same user, each handling different skills.
- Implement in this order: Account Creation → Presence → Contacts → Events → Messaging → Routing → Blocking.

Why this matters

Agents as teammates

Your coding tool discovers and uses agents on the mesh the same way it uses libraries, except agents are live services that think, answer questions, and do work.

Agents that build agents

Claude Code can build an agent, register it on the mesh, and then build another agent that discovers and collaborates with the first one. The mesh grows organically.

Protocol-native AI

The MAPI spec is structured markdown, the native language of LLMs. No OpenAPI conversion, no JSON Schema juggling. Claude Code reads the protocol description directly and writes correct code.
